# Classical baselines for black-box LLM optimization

Standalone-инфраструктура для прогона классических методов оптимизации
первого порядка и BFGS на стандартных тестовых функциях, с единым
форматом логирования и метриками, совместимыми с прогонами
LLM-оптимизаторов из основной части проекта.

Назначение модуля — обеспечить корректную точку отсчёта для сравнения:
любой LLM-прогон может быть напрямую сопоставлен с классическим методом
на той же функции, при той же стартовой точке и при том же бюджете итераций.

## Требования

- Python 3.10 или выше
- numpy, scipy, matplotlib

```bash
pip install numpy scipy matplotlib
```

## Структура

| Файл | Содержание |
|---|---|
| `benchmarks.py` | Пять тестовых функций (Sphere, Rosenbrock, Rastrigin, Ackley, Griewank) с аналитическими градиентами и известными минимумами; обёртки `ShiftedBenchmark` и `RotatedBenchmark` для трансформаций, сохраняющих свойства функции. |
| `classical.py` | Реализации GD, Heavy Ball, Nesterov accelerated gradient, Adam, AdamW и BFGS под единым интерфейсом; процедура подбора learning rate через grid search. |
| `llm_interactive.py` | Интерактивный LLM-оптимизатор с промптом V1 (`x`, `f(x)`, `grad f(x)` в истории); поддерживает replay-режим для пересчёта метрик без повторного обращения к модели. |
| `metrics.py` | Расчёт метрик траектории: финальный и наилучший gap до оптимума, время до ε-точности, success rate, длина траектории, косинус шага с антиградиентом, согласованность последовательных шагов. |
| `runner.py` | CLI-обёртка с четырьмя подкомандами: `tune-lr`, `run-classical`, `run-llm`, `compare`. |
| `plots.py` | Сводные визуализации: сетка кривых сходимости, сетка траекторий на 2D-контурах, heatmap-таблицы метрик. |
| `configs/scenarios.md` | Готовые сценарии запуска для типичных задач. |

## Интерфейс оптимизаторов

Все методы соответствуют единой сигнатуре:

```python
xs, fs = optimizer(f, grad, x0, K, **hparams)
```

где `xs` — массив траектории формы `(K+1, n)`, `fs` — массив значений функции
формы `(K+1,)`, включая начальную точку. Этот же интерфейс реализуют
LLM-оптимизаторы, что позволяет применять одинаковую процедуру логирования
и расчёта метрик к любым реализациям.

## Подбор learning rate

Скрипт `tune-lr` выполняет grid search по сетке `lr` из 21 точки,
логарифмически распределённой в диапазоне от 1e-5 до 1.0. Каждый метод
запускается на каждой стартовой точке; в качестве итогового значения для
метода выбирается медианный по `J` стартам learning rate, минимизирующий
финальное значение функции. Медианный выбор устойчив к выбросам,
возникающим при расходимости с большими шагами.

Запуски, приводящие к переполнению или к нечисловым значениям, отбрасываются
и не учитываются при выборе lr.

## Метрики

Для каждой траектории вычисляются:

- **Финальный gap** $\Delta_K = f(x_K) - f^{*}$.
- **Наилучший gap** $\min_k f(x_k) - f^{*}$ — устойчив к нестабильным методам, теряющим достигнутые значения.
- **Время до ε-точности** $T_{\varepsilon} = \min\{k : f(x_k) - f^{*} < \varepsilon\}$.
- **Success rate** — доля стартов, на которых $T_{\varepsilon}$ достигнут в пределах бюджета.
- **Косинус с антиградиентом** $\cos(\Delta x_k, -\nabla f(x_k))$ для каждого шага и среднее значение по траектории.
- **Согласованность шагов** $\cos(\Delta x_k, \Delta x_{k-1})$ — индикатор устойчивого направленного движения по сравнению с осцилляциями.
- **Длина траектории** $\sum_k \|x_{k+1} - x_k\|$.
- **Размеры шагов** $\|x_{k+1} - x_k\|$ — медиана и максимум.

## Типовой запуск

```bash
# Подбор learning rate для всех методов на сетке функций × размерностей
python runner.py tune-lr --dims 2 5 --J 10 --K 50

# Прогон полного протокола: все функции, обе размерности, J стартов, K шагов
python runner.py run-classical --dims 2 5 --J 10 --K 50

# Свод результатов: сводная таблица + графики
python runner.py compare
```

После выполнения в директории `logs/`:

- `tuned_lrs.json` — оптимальные learning rate по медиане стартов;
- `classical_runs.json` — все траектории классических методов с метриками;
- `combined_runs.json` — объединённый файл, включающий LLM-прогоны при их наличии;
- `summary_table.csv` — агрегированная таблица для импорта в отчёты;
- `fig_convergence_grid.png` — кривые сходимости $f(x_k) - f^{*}$ в логарифмической шкале;
- `fig_trajectories_2d.png` — траектории на $(x_1, x_2)$-плоскости поверх контуров целевой функции;
- `fig_metrics_heatmap.png` — heatmap значений метрики по парам (функция, метод).

## LLM-оптимизатор

Запуск интерактивного LLM-цикла:

```bash
python runner.py run-llm \
    --model claude \
    --function rosenbrock \
    --n 2 \
    --start-idx 0 \
    --K 30
```

Скрипт на каждом шаге выводит в консоль готовый промпт V1, ожидает ответа
модели в формате `[float, float, ...]`, парсит результат и продолжает цикл.
Парсер устойчив к различным форматам ответа и допускает до трёх повторов
при ошибках разбора; неуспешные шаги отдельно фиксируются в логе.

Все промпты, ответы модели и вычисленные точки сохраняются в JSON-лог по
ходу выполнения, что обеспечивает воспроизводимость и устойчивость к
прерываниям. Сводная запись для соответствующего прогона имеет тот же
формат, что и записи классических методов, и автоматически подхватывается
командой `compare`.

Для повторного расчёта метрик по сохранённой траектории без обращения к
модели предусмотрен replay-режим:

```bash
python runner.py run-llm --model claude --function rosenbrock \
    --n 2 --start-idx 0 --K 30 \
    --replay logs/llm_claude_rosenbrock_n2_start0.json
```

## Стартовые точки

Стартовые точки выбираются равномерно в области определения функции и
сдвигаются от начала координат, если оказываются слишком близко к нему.
Сдвиг от нуля учитывает эффект, обсуждаемый в Huang et al. (2024):
языковые модели смещены в сторону начала координат и могут демонстрировать
ложные «успехи» при стартах вблизи нуля, особенно на функциях с минимумом
в нуле. Фиксированный seed обеспечивает воспроизводимость выборки стартов
между прогонами разных методов.

## Трансформации функций

Модуль `benchmarks.py` экспортирует две обёртки, сохраняющие топологические
свойства целевой функции (выпуклость, форму оврагов, расположение
локальных минимумов друг относительно друга), но изменяющие положение
глобального минимума:

```python
ShiftedBenchmark(base, shift)              # tilde_f(x) = f(x - c)
RotatedBenchmark(base, Q, shift)           # tilde_f(x) = f(Q (x - c))
```

где `Q` — ортогональная матрица (получить случайную можно через
`random_orthogonal(n, rng)`). Обёртки реализуют тот же интерфейс, что и
исходные `Benchmark`-объекты, и подходят для подстановки в любые
имеющиеся прогоны.
